In short

Gergely proposes a LangGraph-orchestrated, multi-phase clinical reasoning pipeline with versioned Postgres state, structured JSON outputs at every stage, and LLM-as-judge quality gates that block advancement until checks pass. The architecture walkthrough covers all five pipeline layers — from evidence intake through deliverable generation. After Phase 1 you hold a working, gated single-phase reasoning loop you can run and inspect before any further commitment.

Architecture I'd propose

  • Evidence Intake — FastAPI + Python ETL
  • Versioned State — Postgres — immutable checkpoints
  • LangGraph Orchestrator — Phase-gated agent sessions
  • Quality Gate — LLM-as-judge, pass/fail threshold
  • Deliverable Generator — Structured JSON output
Assumes: Assumes evidence intake is document-based (PDF or similar); structured DB sources change the ETL layer only · Assumes Postgres/Supabase for state store — swap to another RDBMS without touching orchestration logic · Assumes gold-standard reference outputs exist or can be co-authored in Phase 1 — tell me where I'm wrong and the design adapts.

Plan

  • Phase 1 — Gated loop (Week 1-2) — Working single-phase reasoning loop with quality gate
  • Phase 2 — Full pipeline (Week 3-4) — All phases wired, versioned state, eval harness
  • Phase 3 — Handoff docs (Week 5) — Engineering-spec architecture docs, failure modes
