In short

Gergely proposes building each of your six workflow groups as an independent, importable Make.com scenario — covering product processing, social publishing, RFQ handling, AI qualification, and offer delivery. Every scenario ships with a .json blueprint and a variable map. The architecture keeps workflows isolated so a break in one never cascades to the others.

Architecture I'd propose

  • REST API triggers — Webhooks + HTTP modules
  • AI enrichment — OpenAI structured output
  • Conditional router — Score / approval filters
  • Meta API — FB / IG / publish
  • RFQ & offers — Schedulers + reminders
  • Error handlers — Retry + failure alerts
Assumes: Assumes your platform exposes stable REST endpoints with documented auth · Assumes approval status is a writable field the Meta integration can poll · Assumes messaging flows use Meta Cloud API — tell me where I'm wrong and the design adapts.

Plan

  • Phase 1 - Core scenarios (Week 1-2) — Product processing + social publish blueprints, live and tested
  • Phase 2 - Messaging flows (Week 3-4) — Messaging, RFQ handling, AI qualification scenarios
  • Phase 3 - Polish & docs (Week 5) — Offer delivery, full blueprint set + variable-map docs
