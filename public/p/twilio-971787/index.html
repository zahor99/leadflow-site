Proposal walkthrough

Per-subscriber Twilio numbers with live CRM caller context

Twilio

In short

Each new subscriber gets a dedicated Twilio number provisioned automatically. When a lead calls that number, the system looks them up in the CRM, pulls what listings they viewed, and delivers that context to the agent as a whisper before the call connects — then forwards seamlessly to their real number. Built on Twilio REST API, TwiML, and a Postgres number-to-agent mapping layer.

Architecture I'd propose

  • Sign-up trigger — Provision number via REST API
  • Postgres mapping — number → agent → real number
  • Inbound call webhook — fires on every inbound call
  • CRM lookup — caller → listings viewed
  • Twilio whisper — context read to agent only
  • TwiML <Dial> — forward to agent's real number
Assumes: Assumes CRM exposes a lookup endpoint by phone number · Assumes agent's real number is stored at subscription time · Assumes whisper delivery; screen-pop is an alternative if CRM supports it · Assumes Postgres or Supabase for the number mapping store — tell me where I'm wrong and the design adapts.

Plan

  • Phase 1 - Core call flow (Week 1-2) — Working number provisioning + whisper + forward for one subscriber
  • Phase 2 - CRM integration (Week 3) — Live CRM lookup, listings context, missed-call voicemail fallback
  • Phase 3 - SaaS wiring (Week 4) — Multi-tenant provisioning, subscription cancellation release, SMS layer
